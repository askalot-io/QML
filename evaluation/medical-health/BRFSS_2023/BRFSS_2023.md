# BRFSS 2023: Declarative Conversion Analysis

**Source:** Centers for Disease Control and Prevention (CDC), Behavioral Risk Factor Surveillance System (BRFSS) 2023 Questionnaire, 127+ pages
**QML Files:** `evaluation/reference-questionnaires/BRFSS_2023/`
**Date:** 2026-03-29

## Objective

Transform the 2023 BRFSS CATI questionnaire (127+ pages, dual-frame landline/cell phone design with 15 core sections, 1 emerging core section, 32 optional modules, and an asthma call-back script) into a declarative QML representation, then run the Z3-based formal validator to detect structural problems hidden in the imperative GOTO-based version.

## Methodology

1. **PDF preprocessing:** Extracted to text using `pdftotext -layout -nodiag` (5,315 lines). Section boundaries identified via grep.
2. **Question inventory:** 274 nodes catalogued across 52 sections (2 intros, 15 core, 1 emerging core, 1 closing, 32 modules, 1 call-back) using 9 parallel extraction agents, verified by an independent judgement agent.
3. **Declarative QML conversion:** 23 standalone section files generated by 9 parallel conversion agents. GOTO routing converted to preconditions; filter nodes absorbed into block/item preconditions; cross-section variables declared as extern with domain constraints.
4. **Formal validation:** Z3 SMT solver run on all 23 files. Two response code bugs found by judgement agent and corrected before final validation.

## Problems in the Original BRFSS 2023 (Exposed by Declarative Conversion)

### P1: Sex Determination Distributed Across Five Variables

**Severity:** LOW
**PDF evidence:** Pages 5–16 (Introductions), Pages 61–67 (Modules 5–6), Page 98 (Module 21)

**Problem:** Sex determination spans five variables (LL09, LL10, CP05, CP06, MSAB.01) across three separate questionnaire sections. The CATI system synthesizes these at runtime into a single canonical sex value. No valid interview path leaves sex unresolved — if LL10/CP06 yields DK or Refused, the interview terminates. However, the absence of a single authoritative variable creates a distributed state that must be unified externally.

**In the imperative version:** The CATI system transparently consolidates the five variables. Cancer screening modules (5 and 6) reference the synthesized value for their sex-based entry gates.

**In the declarative version:** The conversion must collapse these into a single `sex_at_birth` extern variable with domain constraint `{1, 2}`. This is architecturally correct — the original instrument guarantees resolution — but the distributed origin means the extern declaration is a translation artifact bridging five source variables into one.

### P2: Dual-Frame Introduction Produces Non-Unified Respondent State

**Severity:** LOW
**PDF evidence:** Pages 5–16 (Landline and Cell Phone Introductions)

**Problem:** The landline and cell phone introductions are structurally parallel but capture different variables. The cell phone path collects `has_landline` (CP11) and `household_adults` (CP12), while the landline path collects `residential_numbers` (CDEM.09-10). These variables exist only in their respective interview modes, creating a bifurcated state space that later sections must accommodate.

**In the imperative version:** CATI software maintains mode awareness throughout; interviewers know which path they're on.

**In the declarative version:** The `interview_mode` extern variable bridges this gap, but it creates a phantom dependency — a system variable with no respondent-facing question. Demographics section CDEM.09-10 must carry an explicit `interview_mode == 1` precondition.

### P3: Alcohol Binge Drinking Threshold is Sex-Dependent at Runtime

**Severity:** MEDIUM
**PDF evidence:** Page 40, CALC.03

**Problem:** CALC.03 asks "how many times did you have [X] or more drinks on an occasion?" where X=5 for men and X=4 for women. The threshold is injected by the CATI system at runtime based on the respondent's sex. In the declarative QML, the question text is static — a single item cannot change its wording based on prior answers.

**In the imperative version:** CATI text substitution handles this seamlessly.

**In the declarative version:** The QML models CALC.03 as a single Editbox. The sex-dependent threshold cannot be encoded in the question text or in a postcondition without creating two separate items (one for each sex), which would double the item count for what is functionally the same question. The threshold asymmetry is documented but not enforceable.

### P4: Module Selection is External to the Questionnaire

**Severity:** LOW
**PDF evidence:** Page 48 (Closing Statement / Transition to Modules), Pages 49–127 (all 32 modules)

**Problem:** Each of the 32 optional modules is independently selected by each state. The BRFSS questionnaire itself contains no routing logic that determines which modules are administered — this is an external configuration. In the declarative conversion, each module's entry filter references only respondent-level variables (age, sex, prior diagnoses), but the module-level "is this module active?" gate is entirely outside the questionnaire's formal specification.

**In the imperative version:** The CATI system's module selector handles activation.

**In the declarative version:** Each module is a separate QML file. Whether to load a file is a system-level decision. This is architecturally clean but means the Z3 validator cannot verify cross-module routing — each module validates in isolation.

### P5: MCCS.05 DK-Path Creates an Unmatchable Routing Target

**Severity:** LOW
**PDF evidence:** Page 69, MCCS.02 and MCCS.05

**Problem:** In the colorectal cancer screening module, MCCS.02 routes DK responses to MCCS.05 ("How long since your most recent colonoscopy or sigmoidoscopy?"). In the CATI version, "Don't Know" (code 7) is a valid response that triggers this routing. In the declarative QML, DK/Refused are excluded from response labels, so MCCS.02 can never produce the value that would route to MCCS.05. The item exists in the QML but its triggering path is unrepresentable without including DK as a response option.

**In the imperative version:** The DK code is a legitimate routing trigger captured by the CATI system.

**In the declarative version:** MCCS.05 may appear unreachable depending on how the QML handles the DK-path item. The conversion notes it as a Comment or uses an alternative precondition, but the original intent — asking a vaguer question when the respondent doesn't know which exam they had — cannot be replicated without DK as a first-class response.

### P6: Random Child Selection Requires External Randomization

**Severity:** LOW
**PDF evidence:** Pages 120–122, Module 31

**Problem:** Module 31 (Random Child Selection) requires the CATI system to randomly select one child from the household based on birth order. The questionnaire text reads "The child I will be referring to is the Xth child" — the selection algorithm is external. In the QML, the module captures the selected child's demographics, but the randomization step cannot be modeled.

**In the imperative version:** CATI software generates the random selection and injects it into the interview.

**In the declarative version:** The QML models the questions about the selected child but cannot model the selection process itself. This is a fundamental limitation of static questionnaire specification — dynamic household rosters require runtime support.

## Impact Assessment

The BRFSS 2023 is a well-designed national surveillance instrument. The declarative conversion exposes design decisions that are deliberate trade-offs rather than errors:

- **Permissive response handling:** Zero postconditions across 261 items reflects intentional flexibility — DK/Refused are accepted for every question. This maximizes completion rates but prevents Z3 from detecting answer-level inconsistencies.
- **Modular architecture:** The 32 independent modules validate cleanly in isolation, confirming that each module's internal logic is self-consistent. The trade-off is that cross-module interactions (e.g., cancer survivorship modules 8-10 sharing an entry filter with chronic conditions module 7) cannot be validated holistically.
- **Dual-frame design:** The landline/cell phone bifurcation is architecturally necessary for the BRFSS sampling methodology but creates structural asymmetry that the declarative version must bridge with extern variables.
- **CATI runtime features:** Several BRFSS features (sex-dependent binge thresholds, random child selection, module activation) depend on CATI runtime capabilities that are outside the scope of static questionnaire specification. These are documented as limitations, not defects.

## Conclusion

The BRFSS 2023 converts cleanly to 23 declarative QML files totaling 261 items. All files pass Z3 formal verification: SAT globally, zero cycles, zero unreachable items, zero infeasible postconditions. The six identified problems are design-level observations about the relationship between CATI runtime features and static questionnaire specification — none represent data-loss risks or logical errors. The two cross-check fixes corrected response code transcription errors introduced during conversion, not problems in the original instrument. The BRFSS 2023 is a structurally sound questionnaire whose modular, flat-routing design translates naturally to declarative form.
